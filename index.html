1. そもそも、Git / GitHub とは？

このセクションでは、よく混同されがちな「Git」と「GitHub」の役割の違いを、日常的なモノに例えて解説します。まずはこの2つの関係性をしっかりイメージしましょう。

Git (ギット)

ファイルの「履歴」を管理する道具

  • ✔️ 全自動・履歴管理ノート。「いつ」「誰が」「どこを」変えたか全て記録します。
  • ✔️ 自分のパソコンの中（ローカル）で動くツールです。
  • ✔️ いつでも過去の状態に確実に戻せます。
💡 日常の例え：
あなた個人の「家計簿（記録用ノート）」

GitHub (ギットハブ)

ファイルを「共有・相談」する場所

  • ✔️ 高機能なクラウド作業スペース。Gitの記録をインターネット上で管理します。
  • ✔️ みんなのファイルを一箇所にまとめ、共同作業を可能にします。
  • ✔️ 変更箇所のチェックや相談が画面上で簡単に行えます。
💡 日常の例え：
家族全員で見られる「クラウド家計簿アプリ」

2. どんな「困った！」を解決できるのか？

日常業務でよく起こる「ファイル管理のストレス」。GitとGitHubを導入することで、個人とチーム両方の課題がどのように劇的に改善されるのかを確認しましょう。

導入による「無駄な時間」の削減効果（イメージ）

ファイル探しや先祖返りの復旧に費やしている時間は、年間で見ると膨大です。

3. 先に覚える5つの操作

「何をする操作か」「いつ使うか」「日常のたとえ」を1枚ずつ確認してから、下のワークフローを見ると理解しやすくなります。

📌 操作の全体像（複数人での作業にて）

作業の流れは 初回のみclone でプロジェクトを手元にコピーします。以降は pull で最新を受け取り、 add で記録したい変更を選び、 commit で理由付きで確定し、 push で共有します。

🧺 add（ステージング）

記録したい変更だけを選ぶ

修正が一区切りついたとき

提出する書類だけを提出トレイに入れるイメージ

📝 commit（コミット）

変更をメッセージ付きで履歴に確定

意味のある単位で完了したとき

作業日誌に「何を直したか」を書いて保存するイメージ

📥 clone（クローン）

新しく参加したとき（初回だけ）

プロジェクト一式を最初にPCへコピー

他人の作った資料を丸ごと自分の机に持ってくるイメージ

📨 pull（プル）

作業開始前、または共有前

GitHub上にアップした他の人の最新変更分を自分のPCに取り込む(合体して一つにする)

他人の作った資料と自分の作った資料を合体して一つにするイメージ

🚀 push（プッシュ）

自分の確定履歴を共有先へ送る

レビュー依頼前、作業終了前

日誌の最新ページをチームに配布するイメージ
